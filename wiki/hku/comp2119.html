如何解决一个问题？

  • Recognize problem
  • Build abstract model
  • Design algorithm
  • Analyse the algorithm. 给出算法的 upper bound 和 lower bound.

    • Upper bound: 直观理解：how bad could the algorithm be; worst cost

Time Complexity

Insertion

O(1)O(1)O(1)，计算出 key 后直接插入链表头即可。同时记录 key 在链表里的位置。

Deletion

用双向链表实现的话可以做到 O(1)O(1)O(1)，单向链表则最坏 O(n)O(n)O(n)。

一个比较 tricky 的在于： Insert 之后，我们为每一个插入的数维护它在链表中的位置。例如在上面这个图中（这是个单向链表），我们会为 k2k_2k2​ 这个值维护他的后继，一个指向 k7k_7k7​ 的指针。这前驱后继在 insert 里维护完成后不需要再遍历哈希表来查一遍。

现在我们要删除 k2k_2k2​ 的话，对于单向链表，我们需要 k1k_1k1​ 的后继指针指向 k7k_7k7​，但是由于我们只知道 k2k_2k2​ 的后面是 k7k_7k7​ 但不知道 k2k_2k2​ 的前面是谁，我们还需要重新遍历哈希表的 slot 找出 k2k_2k2​ 的前面是谁，才能完成维护工作。

而如果是双向链表的话，我们同时维护前驱指针和后继指针，这样就 O(1)O(1)O(1) 知道了前面是谁、后面是谁，不需要再遍历哈希表了。因此是 O(1)O(1)O(1) 的。

最坏情况下，搜索是 O(n)O(n)O(n) 的：所有元素都在同一个 slot 里。

平均来看，哈希表的效率取决于哈希函数 h(x)h(x)h(x) 的选取，为了方便分析，我们作出一点假设，称为 Simple Uniform Hashing 假设：

任何元素被 hash 到这 m 个 slot 中的一个的概率相等，且与其他元素如何被 hash 无关。

对于 slot jjj，记 njn_jnj​ 为链表长度，因为 n=∑jnjn=\sum_j n_jn=∑j​nj​，有 E[nj]=nmE[n_j]=\frac nmE[nj​]=mn​

The average running time of searching would be the time required to evaluate the hash function +++ the time to traverse the list.
