Lecture 1. 概述

Principles and Functionality of OS

什么是 OS？

  • a PROGRAM that controls the execution of application program.
  • Two main functions:
    1. resource manager (CPU, memory, storage …)

      处理 conflicting requests 来达成 高效、公平（防止资源一直完全被 high priority 程序占用） 的资源使用

    2. interface between application and hardware.
      • 把软件和硬件隔开 (separate)
      • OS 提供 API (Application Program Interface) 供软件使用来请求调用系统资源，方便开发

OS 的核心服务

  • Allow applications to run on the system
  • Allow running applications to use memory as well as share the memory
  • Allow running applications to interact with each other
  • Allow applications to access and share data that stored in persistent storage

系统是如何提供这些服务的呢？我们关注 How: policies and mechanism, efficiency, hardware support.

Process & Processor Management

  • Process === a program in execution
    • need resources
    • process 终止要回收系统资源

一个典型的系统有很多 process，包括user process, kernel process，并且并行 (concurrently)的运行在一个或多个 CPU core 上

  • 系统的职责：
    1. 如何管理控制这些不同进程？
    2. 当程序要求**使用系统资源 (Restricted Operation)**时，如何操作
    3. Provide Illusion of CPUs (一核模拟多核)
    4. Regain Control of CPUs (防止占用资源过久)
    5. Effective and efficient scheduler

Memory Management

冯诺依曼架构下所有指令和数据都放在 physical memory 里面。

  • OS 目标：
    1. 在内存大小有限的情况下，OS 决定什么东西该放到内存里。

      optimize CPU utilization and computer response to users.

  • OS 职责：
    1. 如何让所有进程共享同一个物理内存？(Share single pool of physical memory)
    2. 控制进程可以读写的内存位置

      进程 A 不能修改进程 B 需要使用的内存，不然会出错

    3. 处理进程所需资源多余物理内存的情况
    4. 空闲内存？没有空闲内存？

Concurrency

  • OS Concurrency

    • Many internal kernel data structures may be updated concurrently by multiple execution logics

  • 现代 OS：多核多线程

  • 系统的职责：

    • What kind of primitives are needed for synchronization?

      We may have to examine different kinds of synchronization and concurrency issues

    • What support do we need from hardware and OS in order to build useful synchronization primitives?

      to work correctly and efficiently

File Management

  • Manages storage disks, responsible for storing any data the user created in a reliable and efficient manner on the disks of the system (save and find)

  • File System …

    • Provides a uniform, logical view of information storage
    • Abstracts physical properties to logical storage unit - file
    • Maps files onto physical media and provides mechanisms for applications to manage and access files

  • 对 OS 而言……

    • 如何维护 persistent storage
    • How to implement? (需要数据结构维护)
    • 降低 System IO Cost (Disk access is slow)

OS 很复杂因为要提供众多服务，还有针对不同硬件、软件提供支持。因此需要合理的 OS Architecture 来解决这个问题。

  • Specify the priority with which each component executes. 决定组建运行的优先级
  • Four Main Architecture:
    • Monolithic
    • Layered
    • Microkernel

所有架构都有的两种 OS 模式

CPU runs at least 2 mode:

  • User Mode & Kernel Mode
    • Kernel
      • CPU running in Kernel Mode when OS kernel executes.
      • Highest priority. Can execute any executable commands.
    • User
      • CPU running in User Mode when a user application is running.
      • Lower priority. Not all commands are executable. (Only a subset of commands available)
      • Ensures that one user program cannot execute instruction that may interfere with operation of other user programs
    • 通过一个 register 里的某一位/某几位来 inform CPU 现在是什么 mode
一个例子：英特尔

例如，在 Intel 的 CPU 里就有 444 个 mode，因此在寄存器里需要 2 bits。

系统调用 System Calls

  • System Calls

    A system call is a mechanism that allows an application program to make a request to the OS for resources/services. Set of system calls is API to services provided by OS

    • Mode switch: When it occurs, OS switches from User mode to Kernel mode to execute corresponding kernel’s system function.
      • Done by instructions that help processes to trap into kernel mode and return-from-trap to the user mode.

各种架构

Monolithic
Monolithic
  • 特点：
    • 所有组件放在一起，组件的组织 不够结构化，没有形成模块 (Module)，没有明确的接口 (Interface)
    • 一坨运行在 Kernel Mode 的屎山
    • 一个组件可以直接与另一个组件交互
    • 数据/数据结构可以直接在组件之间共享
  • 优点：
    • 比较高效
  • 缺点
    • 因为直接与硬件沟通，所以和硬件相关的代码（例如调用硬件指令）直接写死在代码里，散落在各地的代码里，难以维护
    • 因为数据直接共享，A 组件对数据/数据结构的修改可能会影响 B 组件，一旦 出现 bug 可能导致 fatal error；也更容易被病毒攻击（被病毒修改一下系统组件用的数据就寄了）
    • 省流：逐渐屎山
Layered
Layered
  • 特点
    • Groups components that perform similar function or specific role into a module and layers the modules one on top of the other
    • Each layer communicates only with layers immediately above and below it
      • Processes’ requests might pass through many layers before being serviced
        • System efficiency can be less than monolithic kernels
  • 优点
    • 调试、组织更加便利
    • Information Hiding：可以对其他层隐藏一些重要信息，只暴露接口；也使得更新、修改更新方便
    • Easier to Draw Kernel-User Boundary
一个例子：虚拟机
虚拟机分层原理
Microkernel
Microkernel

  • Moves as much functionalities from the kernel space into user space processes (being called as servers)

  • Pros

    • Attempt to keep kernel small
    • The system becomes more stable as only the bare essentials are running in kernel mode
    • More easy to extend and port to other platforms

  • Cons

    • Server processes (in user space) are interacting by means of message exchanges, which need the helps from kernel
      • Induce a significant performance overhead because of communications have to go through kernel
Modular
Modular

Core components are encapsulated as modules. The whole kernel is a collection of modules.

  • More flexible and efficient
    • Protected APIs
    • In kernel →\to→ efficient
    • Allow dynamic module loading
    • Extensiable

Lecture 2. Process Abstraction

What is “process”?

一个运行的程序就是一个进程，但是进程除了要知道我运行的是什么程序以外，还需要知道一些其他信息，例如 Memory, Resource in Use, Execution State 等，才能有效的管理正在运行的程序。

Process (1): Address Space

Process 的 Address Space 是a range of memory locations/addresses，这一个 range 的 memory 都可以为进程所使用。

细分来说分四种：

Stack: 栈空间。局部变量、函数参数、函数调用、返回值等都在栈中临时保存

Heap: 堆空间。保存动态开辟的数据

Data segment: 保存全局变量和静态变量 (static variables)

Text segment: 保存程序代码用以执行

Process (2): Process State

根据进程运行的过程不同，进程的运行分成几种状态：

  1. New (Initial): 进程刚刚被创建
  2. Running: 进程正在运行
  3. Blocked: 进程被阻塞。通常是进程在等待某些资源被其他进程释放、或者正在等待 event 结束 (例如 I/O event)
  4. Ready: 进程准备可以运行了。正在等待被分配到一个 Processor
  5. Terminated (Final/Zombie): 进程运行完毕，但是还没被清除
Process State 可视化

Process Control Block

进程使用数据结构来记录 Address Space 和 Process State (以及还有其他的) 信息，在 OS 里，这样的数据结构叫做 Process Control Block (PCB) 或者 Process Descriptor

PCB 通常包括
  1. Unique Process ID
  2. Process State
  3. Program Counter: the address of next instruction
  4. Register Content: snapshot of the registers contents in which the process was last running before transitioned out of the Running State
  5. Scheduling information - process priority, pointers to scheduling queues, etc.
  6. Credentials - determines the resources this process can access
  7. Memory Management information - concerning memory areas allocated to the process
  8. Accounting information - CPU usage statistics, time limits, etc
  9. A pointer to the process’s parent process
  10. Pointers to the process’s child processes
  11. Pointers to allocated resources
  12. …

Process Table

OS 用来快速查找一个进程的 PCB，Proess Table 的每一行都是指向一个 PCB 的指针。

Process Table 原理

当进程完全结束时，OS 从 Process Table 删除这个进程的 PCB 指针，然后释放进程之前占用着的所有资源。

Process List Structures

先前提到过，进程可以从 Blocked 状态和 Ready 状态转移到 Running 状态。OS 在内部维护着一个 Ready List 和 Blocked List 来保存进程的引用

  • 当父进程被 terminate 的时候，现代操作系统允许子进程独立于父进程继续运行

读写冲突

  • 并发的问题：如果同时读写数据，输出会不一致。
  • 原因： Uncontrolled Scheduling, can’t predict when and how long a thread will execute.
  • 我们把多个线程 同时 访问和修改共享的数据，其输出由执行顺序 (a race) 决定的情况称为 Race Condition.
例子

初始 x=10，在 Thread 1 和 Thread 2 里都执行 x++;，结果即可能是 111111 也可能是 121212.

为 111111 时就是这种情况：

Case 1
Critical Sections

把需要修改共享数据的代码称为 Critical Section，这些代码在修改数据时需要 protection

  • 仅仅读取数据时，允许多个线程一起读。
  • 但是只能有一个线程写入数据，正在写入数据的线程称为 in its critical section. 显然任意时刻，我们希望最多只有一个 thread 修改数据。
  • Mutual Exclusion.
解决办法: Atomic Operation

【定义】The operation can not be interrupted in the middle, and hardware can guarantee either has done or not done at all.

Mutual Exclusion

We need a mechanism to inform OS that a thread is going to enter its critical section

  • A lock is a data structure used for indicating the start and end of a critical section.
    • Indicate shared data is about to be accessed and ask for the system to provide necessary protection
  1. requires the lock, wait until gets the lock
  2. Do something to the data
  3. Release the lock

但是这个流程只在 Single Core 或者在 Kernel Mode 里才有效。

实现锁的两种 Atomic Operation

Atomic Operation 实现两种逻辑:

Test-and-Set

  • enables us to test the old value while simultaneously set the variable to a new value
  • The instruction returns the old value and simultaneously updates the variable to new value in an atomic way
  • Checking the value of a shared variable to determine whether it can enter critical section.
  • Uses busy waiting (spin-lock, spin-wait) to test whether can enter.
  • 问题
    • processor 一直运行死循环，资源浪费。
    • 无法避免无限拖延 infinite postponement
    • 没有 fairness guarantee

逻辑是：如果当前的锁没有被释放，flag 的值应该是 111；否则如果在某个时刻被释放了，TestSet() 会捕获到 tmp=flag=0 然后将 flag 重新设置为 111 进入占用模式。而 TestSet() 返回旧的值（tmp），于是退出 while() 循环，开始执行代码。直到 flag=0 重新释放锁。

Compare-and-Swap

  • 测试 value 的值和 expected 是否一致;
  • 是的话，替换为新的值，返回当前的值
  • 否则啥也不干

Producer-Consumer Model

消费者、生产者问题

  1. Buffer 是有限且共享的
  2. Producers 生产数据并放入 Buffer
  3. Consumers 接收数据并从 Buffer 里移除

  • A consumer must not consume an empty buffer when buffer pool is empty

Lecture 7. Deadlock 死锁

【问题描述】A set of threads each holding some system resources and block waiting to acquire another system resource held by another thread in the set.

死锁的触发条件：

  1. MutEx：最多只能有一个线程独占资源
  2. Wait-for Condition (Hold and wait condition)
    • A thread may hold some resources while awaiting assignment of additional resources
  3. No-preemption condition: 资源的控制权不会被系统从线程手里收回
  4. Circular-wait condition: 循环等释放

Prevention

用 Restrictive Policy 来 remove 上述四个条件中的任意一个即可破除死锁。

【破 Circular-Wait Condition】

给每一个 resource 标号，规定每个线程按所需要的资源的标号从小到大获取、执行。

劣势：线程可能先需要大标号的资源然后才是小标号的资源，这样会导致资源的利用率降低。

【破 Hold-and-Wait Condition】

线程开始时，要么一次性给予所有资源，要么什么也不给然后挂起，等到之后再试一次。

劣势：

  1. 资源利用率低；
  2. 需要很多资源的线程会等待太久

【破 The Mutual Exclusion condition】

Sharable resources, if allow non-mutually exclusive access, do not result in deadlock

【破 No-preemption condition】

如果线程 AAA 需要的某个资源被占用，那么 AAA 就必须释放所有已占有的资源，然后挂起；

当且仅当 AAA 能够一次性获得所有需要的资源（旧的和新的），才重启继续运行。

劣势：

  1. Substantial overhead 资源开支太大，因为可能丢失之前的状态、数据
  2. Starvation 等太久

Deadlock Avoidance

并非事先尝试破坏 Deadlock 发生条件的任何一条，而是事先知道进程需要的资源，通过算法规避死锁。

Avoidance requires some global knowledge of which locks/resources various threads might grab during their execution. Subsequently schedules the threads in a way as to guarantee no deadlock can occur

Avoidance by Scheduling

事先获取进程的资源依赖，用依赖关系作为 CPU 分配的 guideline.

  • 但是仍然可能导致低资源利用率

Banker’s Algorithm

  • 允许在索要新资源的时候持有已有资源
  • 当系统估计不会出现死锁时，会将资源分配给进程

特点：

  • 可能会有死锁的情况出现
    • 但是会定时监测死锁是否出现
      • 通常是检查 circular-wait 是否存在（维护一个依赖图，然后检查环）
    • 如果出现，需要终止死锁
      • Method 1, 把死锁的进程全部终止
      • Method 2, 每次结束一个死锁着的进程，直到死锁解除（结束尽可能少的进程，并且解决死锁）
      • 但总会导致一些进程的工作丢失
