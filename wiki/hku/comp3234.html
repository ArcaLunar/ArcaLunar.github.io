IP 欺骗（IP spoofing）是指创建源地址经过修改的 Internet 协议 (IP) 数据包，目的是隐藏发送方的身份或冒充其他计算机系统。这种技术常被恶意用户用于发动分布式拒绝服务 (DDoS) 攻击。

在 IP 欺骗中，攻击者伪造数据包的源 IP 地址，使其看起来像是来自可信的来源。这不仅可以掩盖攻击者的真实身份，还能绕过某些安全措施。例如，攻击者可以利用这种技术发送大量伪造的数据包，导致目标服务器过载并崩溃。

Denial of Service (DoS)
attackers make resources (server, bandwidth) unavailable to legitimate traffic by overwhelming resource with bogus traffic

应对措施：

  • authentication: proving you are who you say you are
    • cellular networks provides hardware identity via SIM card; no such hardware assist in traditional Internet
  • confidentiality: via encryption
  • integrity checks: digital signatures prevent/detect tampering
  • access restrictions: password-protected VPNs
  • firewalls: specialized “middleboxes” in access and core networks:
    • off-by-default: filter incoming packets to restrict senders, receivers, applications
    • detecting/reacting to DOS attacks

Chapter 3. Transport Layer

3.1 【 RoadMap 】 Transport Layer Services and Protocols

  • Logical Communication
    • 在两个通信的 host 之间我们假想出一个直接连接他们的 channel，这个 channel 就是 Logical Communication
    • 有别于 in reality，这两个 host 之间会经过各种 routers and links
    • 经过这样一层 abstraction，我们可以不用关注 routers and links 内部对数据的处理方式，只关注发送和接收这两个部分
  • 与 Network Layer 的区别：
    • Transport Layer 关注 communication between processes
    • Network Layer 关注 communication between hosts
    • processes 运行在 hosts 上。
      • 比喻：host=城市 A,B；process=住在 A 的人 C, 住在 B 的人 D
  • Transport Layer Actions
    • 发送端 Sender:
      • 数据从 Application Layer 通过 Socket 发给 Transport Layer
      • Transport Layer determines Segment Header Fields Value
      • create Segment
      • Pass segment to IP (Network Layer)
      • 等到 Network Layer 把数据发送到接收端 Receiver 后
    • 接收端 Receiver:
      • Receive segment from IP
      • 检查 Header value
      • Extract application layer message
      • Demultiplex message up to applicaiton via Socket
  • 两个重要的协议：TCP 和 UDP
    • TCP: Transmission Control Protocol
      • Reliable, in-order delivery
      • Congestion control
      • flow control
      • connection setup
    • UDP: User Datagram Protocol
      • Unreliable, unordered delivery
    • Services not available
      • Delay guarantees
      • Bandwidth guarantees

3.2 Multiplexing and De-multiplexing

实际上，每一层都有 multiplexing 和 demultiplexing.

我们来考察 UDP 协议下的 (de-)multiplexing。假定 Client1(p3) <- Server(p1), Server(p2) -> Client2(p4)

  • Multiplexing at Server
    • 处理从不同 Socket (p1, p2) 传递下来的数据，添加 Transport Header (之后也会用于 de-multiplexing)
  • Demultiplexing at Server
    • 处理从 p3, p4 接收的数据，利用其中的 Header Info 把 segments 转发到对应的 Socket

3.4 Principles of Reliable Data Transfer (RDT)

我们从理想模型出发，一步一步加入现实因素。我们用 Finite State Machine 表示 sender/receiver 应该采取的 action 和当前的 state

RDT ver 1.0 理想模型

一个理想模型是，sender 和 receiver 之间的 Logical Connection 非常稳定，不会出岔子（指 packet lost, bit corrupt, congestion）。

在这种模型下，我们的 sender 只需要发送数据即可；receiver 只需要接收数据即可.

RDT ver 2.0 Bit Corrupt

现在，Logical Connection 不太稳定，发送的数据里的某些二进制位可能被 flip。

  • 类比现实世界，receiver 用信号标记 sender 发过来的 packet 是不是对的，是的话接受，否则要求重发。
    • 可以使用 checksum 检查是不是对的
    • receiver 给 sender 发送 Acknowledgement，说明 packet 是对的，继续发下一个；否则发送 Negative Acknowledgement
  • 如果 sender 收到 NAK，就再发一遍
  • Stop and Wait 机制
    • 上一个没有安全送达前不发送下一个。

RDT ver 2.1 ACK/NAK Corrupt

如果 ACK 损坏了，我们不能直接重发所有包，有可能重复发送了。所以我们只重发需要的包，需要什么包由 packet 里定义的 sequence number 决定。receiver 舍弃重发的 packet。

RDT 2.1 如果只用 seq_num = 0, 1 的问题

Sender:

  1. 要检查 ACK/NAK 是否损坏
  2. 要记住上一个发的 packet 是 000 还是 111 才能判断接收到的 ACK packet 是不是对应刚刚发出去的那个

Receiver:

  1. 要检查 packet 是否重复

RDT ver 2.2 无 NAK

如果 packet 错误，receiver 不发送 NAK，而是总是 send ACK for last packet received OK，因此必须包含 seq_num 来表明 last packet 是哪一个 packet。

于是 seq_num 相同且重复的 ACK 就代表 NAK。

设立 Timeout 来处理丢包，超时了就重发。

Go-Back-N (GBN) 与 Selective Repeat (SR)

算法演示网站

GBN 算法维护一个滑动窗口，维护 NNN 个连续的、已发送但未被 ACK 的数据包。GBN 里的 NNN 指代 Max Sender Window 的大小。

  • GBN 的 ACK(n) 的意思是 1∼n1\sim n1∼n 的 packet 已确认接收完毕。当接收到 ACK(n) 后才把窗口移动到 n+1n+1n+1。
  • GBN 为这个窗口设立一个 timer，如果 timeout（也就是窗口里的第一个元素没被送到），那么这个窗口里的所有数据都会被重新发送一遍。
  • GBN 里，Max Sender Window Size === Seq Num −1- 1−1.

如果 Windows Size === Seq Num 的话，会有一个问题。假定Windows size = M，发 0∼M−10\sim M-10∼M−1 共 MMM 个 packet，并且成功送到，receiver 发回 ACK=M−1ACK=M-1ACK=M−1

但是对于 sender 来说，由于每次发的 packet 的最后一个数据都是 seq=M−1seq=M-1seq=M−1，因此 sender 无法确定这个 ACKACKACK 是因为这一批的 packet 成功送到要发下一批的 packet，还是这一批的 packet 没有送到要重发。

因此最大为 M−1M-1M−1。

  • SR 算法则与 GBN 不同，ACK(n) 表示 nnn 号 packet 成功收到，各个 packet 的 ACK 之间没有关联
  • out-of-order packets 会被 receiver 缓存
  • 对于 sender 来说，会为每一个 packet 都设立 timer，一旦 timeout 就重发这个 packet.

在 SR 算法里，Window Size 最大为 M2M\over 22M​，即 0∼M2−10\sim\frac{M}{2}-10∼2M​−1。如果超过 M2M\over 22M​，那么考虑极端情况，receiver 收到所有的 packets，但是发送回去的 ACKs 全部丢失。

这时，receiver 的 window 移动到后面的位置，包含了下一个 Seq Num 循环的 000；但是在 sender 看来，因为没有收到 ACK，所以这些包都超时了，因此把 0∼Wsender−10\sim W_{sender}-10∼Wsender​−1 的包又都重新传了一遍。

而 receiver 又收到了 packet 000，receiver 以为是下一个 Seq 循环的 000 号包，但 sender 发送的还是上一个 000 号包，产生了错误。

TCP 建立连接：三次握手

握手两次的问题：b→ab\to ab→a，说明 bbb 收到了 aaa 的东西， 证明了 aaa 的发送能力是没问题的，同时证明了 bbb 的接收能力也是没有问题的。如果到这里就结束了， 那么 aaa 的接收能力和 bbb 的发送能力就没办法保证了。

所以需要三次握手，第三次握手可以携带数据。

  1. Client 发送带有 SYN (SEQ = x)（xxx 是随机的） 标志的数据包，发送到 Server，然后 Client 进入 SYN_SEND 状态，等待 server 的确认
  2. Server 发送带有 SYN, ACK (SEQ=y,ACK=x+1) （yyy 也是随机的） 标志的数据包，发送到 Client，然后进入 SYN_RECV 状态
  3. Client 发送带有 ACK (ACK=y+1) 标志的数据包，发送到 Server，然后 Client 和 Server 都进入 ESTABLISHED 状态，完成 TCP 三次握手。

TCP 断开连接：四次挥手

  1. 第一次挥手：客户端发送一个 FIN (SEQ=x) 标志的数据包 →\to→ 服务端，用来关闭客户端到服务端的数据传送。然后客户端进入 FIN-WAIT-1 状态。
  2. 第二次挥手：服务端收到这个 FIN（SEQ=X） 标志的数据包，它发送一个 ACK (ACK=x+1) 标志的数据包->客户端 。然后服务端进入 CLOSE-WAIT 状态，客户端进入 FIN-WAIT-2 状态。
  3. 第三次挥手：服务端发送一个 FIN (SEQ=y) 标志的数据包->客户端，请求关闭连接，然后服务端进入 LAST-ACK 状态。
  4. 第四次挥手：客户端发送 ACK (ACK=y+1)标志的数据包->服务端，然后客户端进入TIME-WAIT状态，服务端在收到 ACK (ACK=y+1)标志的数据包后进入 CLOSE 状态。此时如果客户端等待 2MSL 后依然没有收到回复，就证明服务端已正常关闭，随后客户端也可以关闭连接了。

只要四次挥手没有结束，客户端和服务端就可以继续传输数据！
