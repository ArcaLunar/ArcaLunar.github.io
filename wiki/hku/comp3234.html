Chapter 1. Introduction

What is the Internet?

  • “nuts and bolts” view: a network of networks

    • A lot of Devices

      • Hosts
        1. end systems
        2. Runs apps

    • Packet switches: 转运数据

      • Routers
      • Switches

    • Communication Links

      • e.g. Fiber, copper

      • Transmission rate: bandwidth

    • Networks

      • Collection of devices, routers, links

    • Protocols

      • Control sending, receiving messages
      • define the format, order of messages sent and received among network entities, and actions taken on message transmission, receipt.

    • Internet Standards

      • Content: RFC (Request for Comments)
      • Organization: IETF (Internet Engineering Task Force)

  • “services” view

    • Infrastructure that provides services to apps
    • Provide API to apps

Chapter 2. Application Layer

2.2 HTTP

网页由很多 object (base HTML-file) 组成，包含 referenced objects，可以被 URL 访问（URL = {hostname}/{path name}）

client: receive Web objects and display them
server: sends Web objects

【流程】 使用 TCP。

  1. client 初始化 TCP 连接，创建 socket，port 80
  2. server 接收 TCP 连接
  3. 交换 HTTP messages
  4. TCP 关闭
  • HTTP is stateless
    • 不保存之前的 request 的状态

HTTP Techniques

Two types of HTTP
  1. Persistent
    • 创建 TCP
    • 多个 object 一次性传输
    • 关闭
  2. Non-persistent
    • 创建 TCP
    • 一次只传输一个 object
    • 关闭
    • Total time === 2×2\times2× RTT + file transfer time.

RTT, Time for packet to travel from client to server and back again.

HTTP messages

【 HTTP Request 】

HTTP message

(1) POST

  • User input

(2) GET

  • Send data to server

(3) HEAD

  • 只获取 head，而不获取实际文件

(4) PUT

  • 上传

【 HTTP Response 】

Response Code

  1. 200 OK
  2. 301
  3. 400 Bad Request
  4. 404 Not Found
  5. 505 HTTP Version Not Supported
Cookies: 记住 client & server 状态

Component

  1. cookie header line of HTTP response
  2. cookie next header line of HTTP request
  3. cookie file kept on user’s host (managed by browser)
  4. back-end database at Website

HTTP response 返回时也会告诉 client set cookie, id=xxx. 并且在之后的通信里，也会带着 cookie id

Web caches (aka proxy server)

client 在发送给 server 前先发到 cache，没找到再发给 server.

  1. Reduce response time
  2. Reduce traffic
  3. 差带宽也可以有好体验

【 Cache 相关的计算 】

设 cache hit rate =0.4=0.4=0.4, access link =1.54Mbps=1.54Mbps=1.54Mbps, LAN =1Gbps=1Gbps=1Gbps, download =1.5Mbps=1.5Mbps=1.5Mbps

  • Over access link: 0.6×1.5/1.54=0.580.6\times 1.5/1.54=0.580.6×1.5/1.54=0.58 →\to→ lower queueing delay
另一种形式的 caching: Conditional HTTP GET

有最新的 copy 就不 send.

  • 在 HTTP Request 里维护 If-modified-since: <date>
    • 如果是最新的，返回 HTTP 304 Not Modified
    • 否则 HTTP 200 OK + object

HTTP/2 and HTTP/3

  • HTTP/2: 和 HTTP/1.1 变化不大

    • Transmission order specified by client.
    • push unrequested object to client (for future use)
    • divide objects into frames, which can be scheduled to mitigate HOL blocking。速度快的先处理

  • HTTP/3:

    • congestion control
    • packet loss

2.3 Email, SMTP, IMAP

  • User Agent
  • Mail server
    • mailbox: 发送给 user 的邮件
    • message queue: user 发送给别人的邮件
  • SMTP
    • 把邮件 distribute 给 user
    • 转发邮件到其他 mail server
    • 每个 host 可以同时充当 client 和 server

SMTP 的特点

  1. 使用 TCP 建立可靠的连接
  2. 固定端口 Port 25
  3. 所有消息都被编码成 7-bit ASCII

【例子】 邮件接受与发送

【邮件发送】 SMTP 流程

  1. client 与 server 之间建立 TCP 连接 (333 次握手)
  2. server 向 client 发送 220
  3. client 向 server 发送 HELLO
  4. server 向 client 发送 250 HELLO。SMTP 333 次握手结束，开始接收邮件信息。
  5. client 发送 MAIL FROM
  6. client 发送 RCPT TO
  7. DATA 发送的数据
  8. QUIT 结束。

【邮件接收】 IMAP: 从 server 处获取邮件

  • POP 协议：Post Office Protocol. 授权后才能下载
  • IMAP: Internet Mail Access Protocol.
  • HTTP.

SMTP 与 HTTP 比较

【SMTP 和 HTTP 比较 】

  1. HTTP: client pull. SMTP: client push
  2. 都是用 ASCII 编码
  3. 不同点：
  • HTTP: 一个 object 封装在一个 response message 里
  • SMTP: 一个 response message 可以包含多个 object

Chapter 3. Transport Layer

3.1 【 RoadMap 】 Transport Layer Services and Protocols

  • Logical Communication
    • 在两个通信的 host 之间我们假想出一个直接连接他们的 channel，这个 channel 就是 Logical Communication
    • 有别于 in reality，这两个 host 之间会经过各种 routers and links
    • 经过这样一层 abstraction，我们可以不用关注 routers and links 内部对数据的处理方式，只关注发送和接收这两个部分
  • 与 Network Layer 的区别：
    • Transport Layer 关注 communication between processes
    • Network Layer 关注 communication between hosts
    • processes 运行在 hosts 上。
      • 比喻：host=城市 A,B；process=住在 A 的人 C, 住在 B 的人 D
  • Transport Layer Actions
    • 发送端 Sender:
      • 数据从 Application Layer 通过 Socket 发给 Transport Layer
      • Transport Layer determines Segment Header Fields Value
      • create Segment
      • Pass segment to IP (Network Layer)
      • 等到 Network Layer 把数据发送到接收端 Receiver 后
    • 接收端 Receiver:
      • Receive segment from IP
      • 检查 Header value
      • Extract application layer message
      • Demultiplex message up to applicaiton via Socket
  • 两个重要的协议：TCP 和 UDP
    • TCP: Transmission Control Protocol
      • Reliable, in-order delivery
      • Congestion control
      • flow control
      • connection setup
    • UDP: User Datagram Protocol
      • Unreliable, unordered delivery
    • Services not available
      • Delay guarantees
      • Bandwidth guarantees

3.2 Multiplexing and De-multiplexing

实际上，每一层都有 multiplexing 和 demultiplexing.

我们来考察 UDP 协议下的 (de-)multiplexing。假定 Client1(p3) <- Server(p1), Server(p2) -> Client2(p4)

  • Multiplexing at Server
    • 处理从不同 Socket (p1, p2) 传递下来的数据，添加 Transport Header (之后也会用于 de-multiplexing)
  • Demultiplexing at Server
    • 处理从 p3, p4 接收的数据，利用其中的 Header Info 把 segments 转发到对应的 Socket

3.3 Connectionless Transmission: UDP

UDP

类比：网络漂流瓶

  • “best effort” service, UDP Segments may be lost or delivered out of order

  • Connectionless

    • No handshaking

  • 应用场景

    • 如果需要使用 UDP 并且要 reliable transfer，那么应该在 Application Layer 控制

3.4 Principles of Reliable Data Transfer (RDT)

我们从理想模型出发，一步一步加入现实因素。我们用 Finite State Machine 表示 sender/receiver 应该采取的 action 和当前的 state

Step 1. 理性模型

一个理想模型是，sender 和 receiver 之间的 Logical Connection 非常稳定，不会出岔子（指 packet lost, bit corrupt, congestion）。

在这种模型下，我们的 sender 只需要发送数据即可；receiver 只需要接收数据即可.

Step 2. Bit Corrupt

现在，Logical Connection 不太稳定，发送的数据里的某些二进制位可能被 flip.

我们需要一种机制来检查二进制位有没有被 flip。一种通用的方法是 checksum 方法。在 header 里额外添加检查信息

Step 3. Packet Lost
